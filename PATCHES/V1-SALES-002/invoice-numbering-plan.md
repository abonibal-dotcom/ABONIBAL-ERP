# V1-SALES-002 Invoice Numbering Plan

## Classification

INF.

This file evaluates invoice numbering strategies for the future account-scoped invoice persistence baseline.

## Requirements

V1 invoice numbers must be:

- Unique within `invoices:{accountId}`.
- Generated from the V1 account boundary, not Firebase UID/provider user id.
- Stable after issue and cancellation.
- Human-readable enough for daily operation.
- Compatible with future sync hardening.
- Safe against silent overwrite.

## Options

### Option A: Simple Local Sequential Number

Example:

```text
1
2
3
```

Pros:

- Easy to understand.
- Easy to implement locally.

Cons:

- High collision risk under future multi-device or sync behavior if generated independently.
- Weak audit context without prefix/date/account context.

Decision:

Not recommended as the final V1 policy by itself.

### Option B: Timestamp-Based Number

Example:

```text
INV-20260705-153012
```

Pros:

- Low coordination requirement.
- Simple local generation.

Cons:

- Less friendly for manual accounting.
- Collision is still possible if multiple invoices are generated in the same time slice.
- Does not express an account-local sequence.

Decision:

Acceptable as a fallback component, but not the preferred V1 policy alone.

### Option C: Account-Scoped Prefix And Sequence

Example:

```text
INV-20260705-0001
INV-20260705-0002
```

Pros:

- Human-readable.
- Easy to verify uniqueness inside `invoices:{accountId}`.
- Compatible with account-scoped storage.
- Can later move sequence allocation to a server-side or sync-safe allocator without changing the invoice record contract.

Cons:

- Local-only sequence generation must check for collisions before write.
- Multi-device sync may require a later stronger allocator.

Decision:

Recommended for V1 persistence baseline.

### Option D: User-Entered Invoice Number

Pros:

- Flexible for migrated or external invoice numbers.

Cons:

- High duplicate and formatting risk.
- Requires validation and conflict UX before it is safe.

Decision:

Defer to a later import/custom-numbering mission.

## Recommended Policy

Use an account-scoped prefix plus sequence:

```text
INV-{YYYYMMDD}-{accountLocalSequence}
```

Minimum future generation behavior:

1. Read only `invoices:{accountId}`.
2. Determine the next local sequence for the date/account.
3. Generate an invoice number.
4. Validate uniqueness within the same account key before write.
5. Retry or fail safely on collision.

The invoice number must be generated by the Invoice service, not the UI, so validation and storage remain centralized.

## Collision Policy

If the generated invoice number already exists in `invoices:{accountId}`, the future service should:

- Recompute the next sequence from current stored invoices.
- Retry once or fail with a validation error.
- Never overwrite an existing invoice.

## Future Sync Note

When sync becomes authoritative, invoice number allocation may need to move to a sync/server boundary. The record contract should still keep `invoiceNumber` stable and unique within account scope.
