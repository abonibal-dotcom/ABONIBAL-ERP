The repository has no `functions`, server, or backend directory; no `firebase-functions` or `firebase-admin` dependency; and no Functions target in Firebase configuration. The TEST configuration points only to `database.test.rules.json` and the Database emulator.

A read-only CLI attempt to list deployed TEST functions failed, so this audit does not claim knowledge of any unversioned remote function. No remote service can be treated as an approved capability until it is represented, reviewed, tested, and deployed from this repository.

## Recommended Firebase Foundation

Add a future TEST-only Cloud Functions for Firebase v2 callable runtime using the Admin SDK. This is a new trusted backend foundation, not a small client transport change.

The callable function is preferred because available Firebase Auth and App Check tokens are attached by the Firebase callable protocol. The handler must reject missing authentication and must not trust client-supplied identity, totals, lifecycle status, numbering, or allocation results.

Recommended future components:

- isolated `functions/` package and lockfile policy;
- `firebase.test.json` Functions and emulator configuration without changing production default behavior;
- callable command endpoint for trusted commercial requests;
- Admin Auth/Firestore/RTDB access with least-privilege service identity where practical;
- Auth, Firestore, Realtime Database, and Functions emulators under one explicit TEST project ID;
- App Check enforcement decision before external production exposure;
- structured safe logging without tokens, raw UIDs, account IDs, or commercial payload dumps.

## Authentication and Account Boundary

Current session resolution uses:

1. Firebase Auth provider identity.
2. Firestore `accountMappings/firebase/providerUsers/{providerUserId}`.
3. An explicit logical `accountId` that must differ from provider UID.
4. RTDB membership mirror `accountMembers/{accountId}/{firebaseUid}` for rules enforcement.

The trusted executor must validate both the authoritative session mapping and current RTDB membership. The request's account ID must match both. Firebase UID remains an authentication/membership key only and is never an operational root.

Because Admin SDK access can have full RTDB privileges independent of ordinary client Rules, trusted code must implement these checks explicitly and use a tightly scoped service account/IAM policy where supported.

## Ordinary Client Permissions

An authenticated account member may eventually:

- read committed own-account commercial state;
- create and revision-update a recorded Return under the current contract;
- submit a trusted execution request;
- read safe command status/receipt for its own account.

An ordinary client must not:

- write an executed Return;
- write return allocation state;
- write trusted receipt results;
- write canonical numbering counters or allocations;
- write commercial group commit markers;
- publish execution-generated `sale_return` movements;
- mutate another account;
- self-provision account membership.

## StockMovement Trusted-Write Boundary

Current TEST Rules allow any authenticated account member to create an otherwise valid StockMovement at an absent ID. That is not sufficient once a movement claims to be a trusted commercial effect.

Future rules and routing must distinguish:

- approved ordinary/manual append operations;
- trusted commercial movements generated by Invoice issue, cancellation, or Return execution.

Recommended policy:

- ordinary clients cannot create `sale_deduction` or `sale_return` records carrying trusted commercial source metadata;
- trusted commercial movement publication occurs only inside the Admin SDK atomic publication;
- client `stockMovements:append` transport cannot dispatch a commercial group member in active trusted mode;
- a future internal action such as `trustedCommercialPublication` may exist inside server code, but no ordinary client capability or transport is registered for it;
- deterministic IDs and append-only immutability remain unchanged.

## Rules-Enforceable Boundaries

RTDB Rules can enforce:

- authenticated account membership;
- account and record path identity;
- denial of direct executed Return writes;
- denial of direct writes to allocation, counters, receipts, and commit markers;
- denial of client commercial StockMovement creation;
- record schema, basic lifecycle, revision, and immutability conditions;
- cross-account and unauthenticated denial;
- membership self-modification denial.

Rules must not be claimed to prove:

- cumulative allocation across all Return records;
- multi-line all-or-nothing business acceptance;
- canonical Invoice status and sold quantities across multiple paths;
- global account/day number allocation;
- exact command checksum semantics and recovery;
- complete commercial group publication.

Those invariants require the trusted runtime.

## Emulator Capability

The future architecture can be exercised comprehensively with the Firebase Local Emulator Suite after adding Functions, Auth, and Firestore configuration alongside the existing Database emulator.

Required emulator scenarios include:

- authenticated callable request and account mapping;
- membership allowed/denied/revoked;
- two concurrent Return execution requests;
- transaction retry and one-winner over-return result;
- multi-line rejection with zero allocation;
- exact and conflicting retries;
- function crash/fault injection between processing stages;
- atomic publication and commit-marker bootstrap;
- malicious direct executed Return and commercial movement writes;
- numbering concurrency and midnight/timezone cases.

Emulator evidence does not replace a separately approved limited TEST deployment gate. Production remains frozen and out of scope.

## Current Security Conclusion

The current client and Rules are correctly conservative for recorded-state sync, but no trusted execution authority exists. `invoiceReturns:update:execute` must remain unregistered until the Functions foundation, allocation transaction, atomic visibility, numbering, rules, emulator, migration, and owner gates all pass.
